**代号：AgentDock（可替换）**
**目标平台：HarmonyOS 6.0+（API 20 起），手机 / 平板 / 2in1（鸿蒙笔记本）三端一套代码**
**文档版本：v1.0 · 2026-07**

## 0. 平台现实核查（交叉验证结论，决定选型的硬约束）

在写架构之前，先把检索交叉验证后确认的平台事实列清楚——这些是不可绕过的工程约束：

| # | 事实 | 对设计的影响 |
|---|------|-------------|
| 1 | HarmonyOS 6.0.0 对应 **API 20**（2025-09 发布），配套 DevEco Studio 6.x；6.0.1 对应 API 21。纯血鸿蒙（NEXT 内核线）**不兼容 Android APK** | 全原生 ArkTS + C++/NAPI 技术栈，无法复用任何 Android 生态库（含 Android 版 llama.cpp 封装、ObjectBox、Room 等） |
| 2 | 鸿蒙官方"一次开发，多端部署"（一多）机制：`deviceTypes` 声明 `phone / tablet / 2in1`，配合三层工程架构（common / feature / product）+ Breakpoint 响应式布局，即可同一 HAP/App 覆盖手机、平板和鸿蒙 PC | 不需要为笔记本单独立项目；但 PC 端需专门适配窗口管理、键鼠、快捷键、侧边栏布局 |
| 3 | 端侧 LLM 推理现状：**llama.cpp 可用 OHOS NDK（clang + musl，aarch64）交叉编译成 .so，经 NAPI 供 ArkTS 调用，CPU 推理是当前唯一对第三方完全开放且可靠的路径**。社区实测鸿蒙笔记本纯 CPU 跑 4B 4-bit 量化模型约 14 token/s。MNN 3.3 亦已官方支持鸿蒙 NEXT，可作备选引擎 | 本地推理引擎选 llama.cpp（GGUF 生态）为主、MNN 为备；模型规格按 CPU 预算设计（0.6B–8B 量化档位） |
| 4 | **NPU 对第三方开放受限**：手机侧 NPU 能力开放走华为认证伙伴/HiAI Foundation 路线，鸿蒙笔记本的 NPU 目前只服务系统功能，Maleoon GPU 的 compute shader 对通用计算不可用 | 架构上预留 NPU/加速后端插槽（MindSpore Lite / HiAI delegate），但**生产路径按纯 CPU 设计容量**，NPU 作为渐进增强 |
| 5 | 系统 AI 能力：HarmonyOS 6 提供智能体框架 HMAF、端侧 RAG API（起始版本 6.0.0(20)）、小艺入口；HarmonyOS 7（API 26）进一步开放 Agent 构建与 A2A 接入 | 自建 RAG/Agent 内核保证可控与离线；同时留出 HMAF 适配层，未来可把 Agent 注册到系统级智能入口作为流量入口 |
| 6 | 后台执行受严格管控：长时任务需 `continuousTask`（有类型白名单），延迟任务走 `workScheduler`（系统按资源择机调度） | "记忆定期整理"必须设计成**可中断、可断点续跑的增量任务**，挂 workScheduler + 充电/空闲条件，不能假设有常驻后台 |
| 7 | 原生 .so 有 musl/TLS 等链接坑（社区已踩过 Go c-shared 的 initial-exec TLS 崩溃案例） | 原生层统一用 C/C++（llama.cpp 本身即 C/C++，无此问题），CI 中加真机加载冒烟测试 |

**结论**：这套系统在鸿蒙 6.0+ 上是完全可落地的，但"生产级"的关键不在功能堆叠，而在于：① 推理层对"本地 CPU 弱算力 + 远程强模型"做统一抽象和自动路由；② 记忆整理等重任务适配鸿蒙后台调度模型；③ 三端 UI 用一多机制一次成型。

## 1. 产品定位与设计目标

一个**个人/团队级智能体工作台**（对标 WorkBuddy 类产品），核心闭环：

> 接入模型（本地 GGUF / 远程 OpenAI 兼容 API）→ 挂载本地知识库 → 定义并运行多个智能体 → 智能体在使用中沉淀记忆 → 系统定期自动整理记忆反哺后续对话。

**设计目标（按优先级）：**

1. **离线可用**：断网状态下，本地模型 + 本地知识库 + 本地记忆完整可用（隐私敏感场景刚需）。
2. **数据不出端**：默认所有知识库、记忆、会话数据只存本地沙箱；远程 API 调用仅发送必要上下文，且逐 Provider 可开关。
3. **三端一致**：手机（单栏）、平板/PC（双栏/三栏）同一套业务代码，交互按端适配。
4. **可扩展**：模型 Provider、Agent 工具、文档解析器、Embedding 后端全部插件化。
5. **生产可运维**：完整的日志、崩溃恢复、数据迁移（schema 版本化）、模型文件校验。

**非目标（v1 明确不做）**：端侧微调/LoRA 训练、多用户账号体系、跨设备分布式推理（预留接口）、Android/iOS 跨平台。

## 2. 总体架构

采用鸿蒙官方推荐的三层工程架构，业务内核与 UI 彻底解耦：

```
┌─────────────────────────────────────────────────────────────────┐
│ products/                          产品定制层（按端差异化）        │
│   default/        手机/平板/2in1 统一入口 (entry HAP)            │
│     - 窗口策略、断点布局装配、快捷键注册(PC)                       │
├─────────────────────────────────────────────────────────────────┤
│ features/                          基础特性层（HAR/HSP 动态共享包）│
│   chat/           会话与消息流 UI + 流式渲染                      │
│   agents/         智能体列表/编辑器/运行面板                      │
│   knowledge/      知识库管理（库/文档/切片/检索调试）              │
│   memory/         记忆浏览、整理报告、人工修订                    │
│   models/         模型管理（本地模型下载/远程 Provider 配置）      │
│   settings/       全局设置、安全、导入导出                        │
├─────────────────────────────────────────────────────────────────┤
│ common/                            公共能力层（纯 ArkTS，无 UI）  │
│   core-llm/       InferenceProvider 抽象 + 路由 + 流式协议归一化  │
│   core-rag/       解析→切片→Embedding→索引→混合检索→重排 流水线   │
│   core-agent/     Agent Runtime（ReAct 循环、工具总线、编排器）    │
│   core-memory/    记忆读写 API + 整理(Consolidation)引擎          │
│   core-data/      RDB 封装、schema 迁移、文件仓库、备份           │
│   core-infra/     日志、事件总线、任务队列、配置、加解密           │
├─────────────────────────────────────────────────────────────────┤
│ native/                            C++ 原生层（NAPI 模块）        │
│   llama_bridge/   llama.cpp 封装：生成、embedding、tokenize       │
│   vec_index/      向量索引（HNSW，基于 hnswlib 交叉编译）          │
│   doc_parsers/    PDF 文本抽取（pdfium/mupdf 二选一）             │
├─────────────────────────────────────────────────────────────────┤
│ HarmonyOS 系统能力                                                │
│   ArkUI / ArkData(RDB=SQLite) / FFRT / workScheduler /           │
│   continuousTask / Asset Store(密钥) / 文件沙箱 / RCP 网络        │
└─────────────────────────────────────────────────────────────────┘
```

**关键架构决策（ADR 摘要）：**

- **ADR-1 推理内核**：llama.cpp（GGUF）为本地主引擎。理由：算子/量化生态最全、纯 C/C++ 交叉编译成熟、embedding 与生成同引擎复用、社区在鸿蒙上有成功先例。MNN 作为第二后端插槽（其对鸿蒙 NEXT 有官方支持，且 NPU 路线打通时切换成本低）。
- **ADR-2 向量存储**：SQLite（鸿蒙 RDB 底层即 SQLite）存元数据 + 自带 FTS 做关键词检索；向量本体与 HNSW 索引文件由原生 `vec_index` 模块管理（hnswlib 编译到 OHOS 极轻量，仅头文件库）。不引入独立向量数据库进程——移动端没有这个条件，也不需要。
- **ADR-3 Agent 运行时自研**：不依赖 LangChain 类框架（无 OHOS 移植），核心就是一个受控的 ReAct 循环 + JSON 工具协议，自研 2k 行以内可控性更好，且与后文上下文治理、长时运行机制深度耦合，自持是必然选择。
- **ADR-4 记忆整理走 workScheduler**：整理任务注册为延迟任务（条件：充电中 + 空闲 + 电量>40%），每次运行做增量批处理并持久化游标，被系统打断可无损续跑。
- **ADR-5 一套 App 包多端分发**：`module.json5` 的 `deviceTypes: ["phone","tablet","2in1"]`，UI 用断点（sm/md/lg）驱动单栏/双栏/三栏切换，PC 特有能力（快捷键、窗口最小尺寸、右键菜单）在 product 层注入。

## 3. 推理层（core-llm）

### 3.1 Provider 抽象

所有模型统一实现同一接口，上层（Agent/RAG/记忆整理）完全不感知本地/远程差异：

```typescript
// common/core-llm/src/main/ets/provider/InferenceProvider.ets
export interface ChatMessage { role: 'system'|'user'|'assistant'|'tool'; content: string;
  toolCallId?: string; toolCalls?: ToolCall[] }

export interface GenerateOptions {
  temperature?: number; topP?: number; maxTokens?: number;
  stop?: string[]; jsonSchema?: object;        // 结构化输出约束
  tools?: ToolSpec[];                          // 工具声明（OpenAI function 格式）
  signal?: AbortSignal;                        // 统一取消语义
}

export interface InferenceProvider {
  readonly id: string;
  readonly kind: 'local' | 'remote';
  capabilities(): ProviderCaps;                // ctx 长度/是否支持工具/视觉/embedding
  chatStream(msgs: ChatMessage[], opts: GenerateOptions): AsyncIterable<StreamDelta>;
  embed(texts: string[]): Promise<Float32Array[]>;   // 本地由 llama_bridge 提供
  countTokens(msgs: ChatMessage[]): Promise<number>;
  healthCheck(): Promise<ProviderHealth>;
}
```

`StreamDelta` 统一为 `{ type: 'text'|'tool_call'|'usage'|'done', ... }`。远程侧把 OpenAI SSE 归一化到该协议；本地侧把 llama.cpp 的逐 token 回调归一化到该协议。**上层永远只写一份流式消费代码。**

### 3.2 本地推理：llama_bridge（NAPI 模块）

```
native/llama_bridge/
├── CMakeLists.txt          # OHOS toolchain, -DGGML_NATIVE=OFF, aarch64 + i8mm/dotprod
├── src/
│   ├── napi_entry.cpp      # NAPI 导出：createSession/generate/abort/embed/tokenize
│   ├── engine.cpp          # llama.cpp 生命周期：模型加载/卸载、KV cache 管理
│   ├── worker.cpp          # FFRT 任务队列：单模型串行、跨请求排队
│   └── stream_cb.cpp       # ThreadSafeFunction 把 token 回调安全抛回 ArkTS 线程
└── third_party/llama.cpp   # git submodule，锁定 release tag
```

工程要点（都是踩过坑的地方，直接按此执行）：

1. **编译**：`-DCMAKE_TOOLCHAIN_FILE=$OHOS_NDK/native/build/cmake/ohos.toolchain.cmake -DOHOS_ARCH=arm64-v8a`；开启 `GGML_CPU_AARCH64` 相关的 dotprod/i8mm 编译分支（麒麟 9000 系与鸿蒙 PC SoC 均支持），关闭 Vulkan/OpenCL 后端（平台不可用）。
2. **线程模型**：推理跑在 FFRT 提交的专用线程池（`ffrt::submit`），线程数 = 大核数（通过 QoS 绑定大核）；token 回调经 `napi_threadsafe_function` 回 UI 线程，避免 ArkTS 侧阻塞。
3. **内存纪律**：模型用 `mmap` 加载（GGUF 原生支持），常驻内存 ≈ KV cache + 计算缓冲。App 设定内存预算表：手机档（12GB RAM 设备）允许 ≤3GB 模型驻留，PC 档（24/32GB）允许 ≤8GB；超预算的模型在 UI 上直接标注"本机不可运行"。
4. **单飞行请求**：本地引擎同一时刻只服务一个生成请求（多 Agent 并发时在 ArkTS 侧排队），embedding 请求可与生成复用同一模型或加载独立小模型（bge-small / bge-m3 量化版，约 100–400MB）。
5. **崩溃隔离**：原生层所有入口 try/catch + 错误码返回，绝不让 C++ 异常穿透 NAPI；模型加载失败/OOM 返回结构化错误，ArkTS 侧降级提示。

**推荐模型档位（GGUF Q4_K_M 基准）：**

| 档位 | 模型规格示例 | 适用设备 | 预期速度 |
|------|-------------|---------|---------|
| Nano | 0.6B–1.5B（Qwen 系小模型） | 中端手机 | 20–40 tok/s，用于记忆整理/标题生成/路由分类 |
| Standard | 4B 量化 | 旗舰手机 / 鸿蒙笔记本 | 8–15 tok/s，日常对话+RAG |
| Max | 7B–8B 量化 | 鸿蒙笔记本（大内存） | 4–8 tok/s，复杂 Agent 任务 |

### 3.3 远程推理：OpenAI 兼容 Provider

- 基于 `@hms.collaboration.rcp`（Remote Communication Kit）实现 SSE 流式（RCP 对流式响应、超时分级、连接复用支持好于老 `@ohos.net.http`）。
- 配置模型：`{ baseUrl, apiKey, modelId, 组织自定义 header }`，天然兼容 OpenAI / DeepSeek / 通义 / vLLM / Ollama / Open WebUI 等一切 OpenAI 兼容端点——这使本 App 可直接充当用户自建推理服务的原生客户端。
- **API Key 安全**：写入 Asset Store Kit（系统级加密仓，绑定设备+应用），内存中仅在请求构造瞬间解密，日志全链路脱敏。
- 重试策略：指数退避（429/5xx），流中断支持从已收 token 处提示续写；网络类型感知（蜂窝网下可配置"仅 Wi-Fi 使用远程大模型"）。

### 3.4 路由器（ModelRouter）

生产级的关键组件——决定"哪个任务用哪个模型"：

```
规则优先级（可在设置中覆盖）：
1. Agent 显式绑定的模型 > 全局默认
2. 任务类别路由：
   - 记忆整理 / 标题 / 摘要 / 意图分类  → Nano 本地模型（省电省钱，离线可跑）
   - RAG 问答（知识库命中）           → Standard 本地 或 用户指定远程
   - 长链条工具调用 / 代码生成         → 远程强模型（若可用），否则本地 Max
3. 降级链：远程失败 → 本地 Standard → 报错并保留可重试上下文
4. 隐私围栏：被标记为"仅本地"的知识库/记忆，其内容禁止进入任何 remote Provider 的 prompt（在 ContextBuilder 层强制过滤，而非靠 UI 约定）
```

## 4. 知识库子系统（core-rag）

### 4.1 数据模型

```
kb_library      知识库：id, name, desc, embed_model_id, privacy_level(local_only|allow_remote),
                chunk_policy(JSON), created_at, stats(JSON)
kb_document     文档：id, lib_id, uri(沙箱路径), title, mime, sha256, size,
                status(pending|parsing|embedding|ready|failed), error, version, updated_at
kb_chunk        切片：id, doc_id, seq, text, token_count, meta(JSON: 页码/标题路径),
                fts 虚表同步（SQLite FTS5，中文用 ICU/jieba 分词方案，见 4.4）
vec_index 文件  每库一个 HNSW 索引文件 + 向量平面文件（float16 存储省一半空间），
                与 kb_chunk.id 建立映射；索引损坏可由向量平面文件全量重建
```

### 4.2 摄取流水线（Ingestion Pipeline）

```
用户导入（filePicker 多选 / PC 端拖拽 / 分享面板接入）
   │
   ▼
[1] 复制入沙箱 files/kb/{libId}/raw/，算 sha256 去重
[2] 解析 Parser（插件注册表按 mime 分发）
      .txt/.md         → ArkTS 直接读
      .pdf             → native/doc_parsers (pdfium 文本抽取；扫描版标记"需 OCR"暂不支持)
      .docx/.xlsx/.pptx→ ArkTS 解 ZIP+XML（纯解文本，无需原生库）
      .html            → 正文抽取（Readability 移植）
[3] 结构化切片：标题感知递归切分（按 H1>H2>段落>句子），默认 512 token/chunk、
    64 token overlap；表格整表成块；策略每库可调
[4] Embedding：批量 32 条/批 → InferenceProvider.embed()
      本地默认：bge-small-zh / bge-m3 量化 GGUF（llama_bridge 复用）
      远程可选：OpenAI 兼容 /embeddings（privacy_level 允许时）
[5] 写入：RDB 事务写 chunk + FTS；向量批量插入 HNSW；更新 doc.status=ready
```

流水线以 **持久化任务队列**（core-infra/TaskQueue，任务表存 RDB）驱动：App 被杀后重启自动续跑未完成文档；每步幂等（以 sha256+step 为幂等键）。大批量导入注册 `continuousTask`（数据传输类型）防止进程被冻结，并在通知栏展示进度。

### 4.3 检索流水线（Query Pipeline）

```
query → [查询改写(可选，Nano模型把口语问题改写为检索式)]
      → 并行：向量检索 topK=24（HNSW, cosine） + FTS5 关键词检索 topK=24
      → RRF 融合（Reciprocal Rank Fusion，k=60）
      → 轻量重排（可选：本地 bge-reranker 量化版；关闭时直接取 RRF 序）
      → 上下文装配 ContextBuilder：
          按 token 预算（默认 2048，随模型 ctx 自适应）贪心填充，
          同文档相邻块合并，附引用元数据 [doc#page]
      → 注入 prompt；回答后 UI 渲染可点击引用（定位到原文块）
```

**为什么必须混合检索**：个人/团队知识库普遍含专业术语、产品型号、标准编号、代码标识符等精确符号，纯向量检索对这类符号召回差，FTS 精确匹配是刚需；反过来口语化提问靠向量。RRF 融合是移动端零成本拿到两者优势的标准做法。

### 4.4 中文分词

SQLite FTS5 默认分词器对中文按单字切分，可用但精度一般。生产方案：在 native 层编译 `simple` tokenizer（开源 SQLite 中文分词扩展，内置 jieba 词库裁剪版）注册进 RDB 的 SQLite 实例；若 RDB 不允许加载扩展（API 限制），退化方案是入库前用 ArkTS 侧 jieba-wasm 预分词、以空格分隔写入 FTS 影子列。两条路都验证过再定，工程上留一周 spike。

### 4.5 知识库管理功能（UI 层 features/knowledge）

- 库级：建库/改名/删除、选择 embedding 模型（换模型触发整库重建向量，走后台任务）、隐私级别、统计面板（文档数/块数/占用空间）。
- 文档级：列表（状态/错误可视）、重新解析、删除（级联清 chunk+向量）、原文预览。
- 调试器（生产级产品与玩具的分水岭）：输入任意 query，展示两路召回、融合得分、最终注入的上下文——运维和调优必备。
- 导入导出：整库打包为 `.adkb`（ZIP：raw 文件 + chunks.jsonl + 向量二进制 + manifest），用于设备间迁移与备份。

## 5. 多智能体子系统（core-agent）

### 5.1 Agent 定义（声明式，存 RDB + 可导出 JSON）

```jsonc
{
  "id": "uuid",
  "name": "行业标准审查助手",
  "avatar": "emoji|uri",
  "systemPrompt": "你是……",
  "model": { "providerId": "local:qwen4b", "temperature": 0.3, "maxTokens": 2048 },
  "knowledge": { "libIds": ["kb1"], "topK": 6, "requireCitation": true },
  "memoryPolicy": { "read": true, "write": true, "scopes": ["user_profile", "agent:self"] },
  "tools": ["kb.search", "memory.recall", "calc", "datetime", "http.fetch?domains=..."],
  "skills": ["builtin-skill-structured"],   // T1.0-10②：挂载技能（skills 表 id，按序）
  "loop": { "maxSteps": 8, "timeoutMs": 120000, "reflection": false },
  "triggers": [],                 // v2: 定时/事件触发
  "uiHints": { "suggestedQuestions": ["..."] }
}
```

#### 5.1.1 Skills 机制（T1.0-10②，2026-07-14 用户指示三件套之二）

**skill = 可复用的指令/能力包**（Markdown 指令正文），与 tools 正交：skill 是「怎么做事」的
知识包，tool 是「能做什么」的执行器（参考 WorkBuddy / OpenClaw / Hermes Agent 的 skill 形态）。

- **定义表**：`skills(id, name, description, content, builtin, enabled, created_at, updated_at)`
  （schema migration #3）。builtin=1 出厂内置不可删、可禁用（与 agent_defs 同护栏）；
  内置种子：结构化回答 / 标注来源 / 先问后答 / 简洁优先（SkillRepo.BUILTIN_SKILL_SEEDS）。
- **挂载**：`agent_defs.skills`（JSON 字符串数组，migration #3 追加列）。Agent 可挂多个，
  **挂载顺序即注入顺序**（作者的表达之一）；挂载项被禁用/删除时如实缺席（不报错不编造）。
- **装配**：注入【稳定前缀】——system 消息尾部按 `## 技能：<名>\n<正文>` 逐块拼接
  （ChatService.systemPromptFrom）。同一 Agent 技能集跨轮不变 ⇒ KV 前缀命中不受影响（R2/红线 11）；
  与 system 同段 ⇒ **anchor 预算自动入账**（anchorTokensOf 取 messages[0]；技能逐字占预算，
  编辑器 anchor>25% 告警随 T1.0-10 完整版覆盖）。
- **UI**：编辑器「技能」分区多选挂载（features/agents）；技能管理面板（SkillManagerPane，
  智能体列表「技能」入口）：新建/编辑/删除 + SKILL.md 粘贴导入/复制导出。
- **交换格式（2026-07 定案）**：agentskills.io 的 **SKILL.md 开放标准**——YAML frontmatter
  （必填 name/description）+ Markdown 正文；Claude Skills / OpenClaw / Hermes Agent 已收敛
  到同一格式，外部生态技能粘贴即用。frontmatter 里 name/description/allowed-tools 之外的键
  导入时忽略并如实告知；标准技能包的 scripts/references/assets 附属资源**不在端侧执行语境内**，
  只装入指令正文。
- **工具声明 allowed-tools（T1.0-10③，migration #4 `skills.allowed_tools`）**：
  技能声明「做事所需的工具」。**挂载技能 = 注入指令 + 并集授予工具**——resolveTools 做
  (Agent 白名单 ∪ 挂载技能声明归一) ∩ 注册表 ∩ Provider 能力，执行点二次校验从同一 allowed
  派生（口径同源）。声明名经别名表（ToolAliases：WebFetch→http.fetch 等）**装配时**归一，
  不在导入时固化 ⇒ 端上每新增一个工具，已导入技能的声明自动获得能力。端上不存在的声明
  （Bash/Read 等系统专用名）如实缺席、永不执行，但**原样保留**（导出无损，别家还能吃）。
  这是三件套 ③「Tools 对齐」的消费侧：预置工具集（kb.search/datetime 已落地，calc/http.fetch
  规划中）每配齐一个，外部技能生态的可用面随之扩大。

### 5.2 运行时：受控 ReAct 循环

```
AgentRun (每次用户消息创建一个 Run，状态机持久化到 RDB)
  state: queued → running → [tool_wait]* → done | failed | cancelled | budget_exceeded

循环：
  1. ContextBuilder 组 prompt：system + 记忆注入(见§6) + RAG 上下文 + 会话窗口(带压缩)
  2. provider.chatStream(..., tools)
  3. 若产生 tool_calls：并行度=1 顺序执行（v1 保守），每个工具有独立超时与参数
     JSON-Schema 校验；结果以 tool 消息回填 → 回到 2
  4. 步数/时间/token 预算任一超限 → 以"预算耗尽"要求模型直接给出当前最优答案
  5. 全程 StreamDelta 直通 UI：文本流式渲染，工具调用渲染为可展开的执行卡片
```

**工具总线（ToolBus）**：工具 = `{ name, jsonSchema, permission, execute(args, ctx) }` 的 ArkTS 注册表。内置工具集：`kb.search`、`memory.recall/append`、`calc`（本地表达式求值，非 eval）、`datetime`、`file.read`（沙箱内）、`http.fetch`（域名白名单 + 用户授权弹窗）。权限模型：工具声明 `permission: none|confirm|always_ask`，敏感工具首次调用弹确认，授权按 Agent 记忆化。

### 5.3 多智能体编排（Orchestration）

v1 提供三种够用且可控的模式，全部建立在"Agent 即函数"抽象上（`runAgent(agentId, input, ctx) → output`）：

1. **Router 模式**：一个入口 Agent（用 Nano 模型做意图分类）把请求分发给专家 Agent，结果直接回传。适合"一个聊天框背后多个专家"。
2. **Pipeline 模式**：编排器按 DAG 顺序执行（v1 限制为线性链），上游输出作为下游输入，可插入人工确认节点。适合"检索→起草→审查"类固定流程。
3. **Agent-as-Tool**：任何 Agent 可被声明为另一个 Agent 的工具（`agent.call:<id>`），由主 Agent 的 LLM 自主决定是否调用。这是最灵活的多智能体形态，且不需要额外编排引擎。

并发控制：全局 Run 调度器限制同时活跃的 LLM 请求数（本地=1，远程=3 可配），子 Agent 调用计入父 Run 的步数与 token 预算，防递归失控（调用深度硬上限 3）。

### 5.4 管理 UI（features/agents）

Agent 列表（运行状态、最近使用）、可视化编辑器（表单式，不做拖拽画布——v1 克制）、运行历史（每个 Run 的完整轨迹回放：每步 prompt/工具入参出参/token 消耗）、导入导出（.json 单体 / 分享给同事）。

## 6. 记忆子系统（core-memory）

### 6.1 记忆分层

```
L0 会话窗口   当前对话的原始消息（RDB messages 表）——不算"记忆"，但为其原料
L1 情节记忆    episodic：每次会话结束（或达阈值）由 Nano 模型生成的结构化会话摘要
              { summary, topics[], entities[], outcome, ts }
L2 语义记忆    semantic：原子事实/偏好条目（系统的核心资产）
              { id, scope, text, kind(fact|preference|decision|todo),
                confidence, source_refs[], embed_vec, created_at, last_seen_at,
                access_count, status(active|archived|superseded_by) }
L3 用户画像    profile：单文档，由 L2 高置信条目定期蒸馏出的紧凑画像（≤500 token），
              直接注入所有允许读记忆的 Agent 的 system prompt
```

scope 设计：`user_profile`（全局）、`agent:<id>`（Agent 私有）、`project:<tag>`（用户手动建的主题域）。Agent 的 memoryPolicy 声明可读写哪些 scope，实现记忆隔离。

### 6.2 写入路径（在线，轻量）

对话进行中不做重活。会话结束事件（切换会话/退后台 5 分钟/消息数阈值）触发一个 **本地 Nano 模型** 的抽取任务：产出 L1 摘要 + 候选 L2 条目（带 JSON Schema 约束输出）。候选条目先进 `memory_inbox`（状态 pending），不直接污染正式记忆——正式化由整理引擎完成。全程离线可跑，这也是坚持内置 Nano 本地模型的核心理由。

### 6.3 定期自动整理（Consolidation Engine）——本系统的招牌能力

**调度**：`workScheduler` 延迟任务，条件 = 充电中 ∧ 设备空闲 ∧ 电量>40%，周期目标每 24h 一次（系统择机）；PC 端因资源宽裕可放宽为插电即可。用户也可手动"立即整理"。任务运行时申请短时 `continuousTask`，并将进度游标（处理到的 inbox id / 阶段号）逐批持久化——**任意时刻被系统杀死，下次从游标续跑**。

**整理流水线（每轮增量执行）：**

```
Phase A  收编 inbox：
  对每条 pending 条目：embed → 在 L2 中检索 top5 相似
    - 无相似(≥0.86 无命中) → 转正 active
    - 高相似 → Nano 模型判定 {duplicate|update|conflict|distinct}
        duplicate → 丢弃，原条目 last_seen_at++, confidence↑
        update    → 新条目转正，旧条目 status=superseded_by
        conflict  → 双方标记 conflict，进"待人工确认"列表（UI 红点）
Phase B  聚类合并：
  对 active 条目做增量聚类（相似度>0.92 的簇），簇内由模型改写合并为单条，
  source_refs 并集保留溯源
Phase C  遗忘与降权：
  score = f(confidence, access_count, age)；低分条目 → archived（不删除，可搜回）
  archived 超过保留期（默认 180 天，可配）→ **仅标记为「可删除」（软删除位），不物理删除**
  物理删除**只能由用户在记忆页显式触发**，走复述式确认（UI 设计规范 [R-48]/[R-48a]）——
  后台 workScheduler 静默物理删除与「记忆可审计、可撤销」的产品立场直接冲突，禁止。
Phase D  画像蒸馏：
  L2 高置信 top 条目 → 重写 L3 profile 文档（diff 展示给用户）
Phase E  产出整理报告：
  本轮 新增/合并/归档/冲突 统计 + 明细，存 report 表，UI 时间线展示
```

**可审计与可撤销**：所有 Phase 的每个变更写 `memory_oplog`（操作日志），用户可在"记忆"页查看每轮报告、逐条撤销（undo 即按 oplog 反向操作）。这是把"AI 自动整理"做成生产级而不是黑箱的关键。

### 6.4 读取路径（RecallService）

Agent 组 prompt 时：L3 画像直接注入（若策略允许）；针对当前用户消息做 L2 向量召回 top5（限定该 Agent 可读 scope，过滤 archived），以 `<memory>` 区块注入并要求模型"仅在相关时引用"。命中即 `access_count++`——使用行为反哺遗忘曲线。

## 7. 数据层与存储（core-data）

```
RDB (关系存储，鸿蒙 ArkData relationalStore，底层 SQLite，开启 WAL)
```

## 9. 安全与隐私设计

1. **密钥**：远程 Provider 的 API Key 存 Asset Store Kit（系统密钥库，硬件级保护），绝不落 RDB/配置文件；导出备份包默认剔除密钥。
2. **隐私围栏（强制执行点在 ContextBuilder）**：`local_only` 知识库与记忆条目在装配上下文时若目标 Provider.kind==='remote' 直接过滤并在 UI 提示"N 条本地内容因隐私设置未发送"。围栏在数据层生效，不依赖任何 UI 逻辑。
3. **权限最小化**：仅申请 INTERNET、必要的文件读（经 filePicker 授权制，不要求全盘权限）；无定位/通讯录等无关权限——也利于过审。
4. **工具安全**：`http.fetch` 域名白名单 + 每 Agent 首次确认；`file.read` 仅限用户显式授权过的沙箱路径；不提供任意 shell/代码执行工具（端侧无沙箱条件，v1 明确不做）。
5. **日志**：分级（debug 仅本地环形缓冲），任何日志路径不落用户内容明文与密钥；崩溃收集默认关闭、用户可选开启。
6. **合规**：上架华为应用市场需隐私声明与数据收集清单；本架构"默认零上传"，声明书好写——这是纯本地路线的额外红利。

## 10. 可观测性与质量保障

- **运行轨迹即日志**：agent_runs/run_steps 表天然是结构化 trace，附每步 token 用量与耗时；设置页提供"诊断包导出"（脱敏 trace + 设备信息）。
- **指标**：本地推理 tok/s、首 token 延迟、RAG 召回耗时、整理任务时长，写入轻量 metrics 表，模型管理页可视化——用户能看到"这台设备跑这个模型值不值"。
- **测试策略**：
  - core-* 层纯 ArkTS 无 UI，单元测试全覆盖状态机（AgentRun、TaskQueue、Consolidation 各 Phase）；
  - RAG 质量回归：内置 30 组"query→期望命中 chunk"金标集（覆盖长文档、表格、术语密集文档三类语料），CI 上跑召回率；
  - 原生层：真机冒烟（模型加载/生成/abort/embed 四用例）纳入每次发版检查单，覆盖 Mate 手机 + MatePad + 鸿蒙笔记本三类真机。

## 17. 三条语音链路（同一 VoiceSession 状态机的三种后端）

**链路 A：级联管线（生产默认，可全离线）**

```
麦克风(AudioCapturer 16k mono) → VAD(Silero, 帧级)
  → 流式 ASR(Zipformer streaming，边说边出字幕)
  → 端点判定(VAD 静音 500–800ms + ASR endpoint 双信号)
  → 文本 → AgentService.runAgent(...)   ← 完整复用 §5：RAG/记忆/工具全部生效
  → 流式回复文本 → 句子切分器(按标点/长度攒句)
  → TTS 逐句合成 → AudioRenderer 播放队列
```

延迟预算（目标"话音落→首音出" ≤ 1.5s，鸿蒙笔记本实测目标 ≤ 1.2s）：端点判定 ~0.6s + LLM 首句(Nano/Standard 本地或远程) ~0.5s + 首句 TTS ~0.15s。关键技巧：LLM 流式输出攒够**第一个短句**立即送 TTS，不等全文；后续句子在播放时并行合成（流水线化）。

**链路 B：远程端到端 Realtime（体验上限档）**

WS 直连，音频帧上行、音频增量下行，模型自带 turn detection 与 tool call。App 侧职责收敛为：音频 IO、工具执行回填、隐私围栏（远程链路禁止注入 local_only 内容）、断线降级——**WS 断开自动无缝切回链路 A**，会话上下文（文本转写）双链路共享，用户无感。

**链路 C：本地端到端（实验插槽，不承诺）**

MNN/Qwen2.5-Omni 或 llama.cpp 音频理解模型。以 `local:omni` Provider 形态接入，跑通即插即用，跑不通不影响主线。排 spike 一周，验收标准：鸿蒙笔记本上 RTF<1 且内存<6GB，达不到就封存等硬件/引擎迭代。

## 18. 电话模式（Call Mode）——语音驱动智能体

### 18.1 会话状态机

```
VoiceSession states:
 idle → listening ⇄ thinking → speaking → listening …（全双工循环）
                    ↘ tool_running（播报"正在查…"过场语，可配置静默）
 任意时刻 barge-in：speaking 中 VAD 检出用户语音 ≥300ms
   → 立即停播+清空 TTS 队列 → 取消未完成 LLM 生成(AbortSignal) → 转 listening
 结束：挂断按钮 / 唤醒词"挂断" / 静默超时(默认 90s)
```

### 18.2 "语音调度智能体 + 强模型接管"——两级大脑模式

这是你要的核心交互，落成一个固定编排（Router 模式的语音特化）：

```
用户语音 → ASR 文本
  ↓
[前台快脑] 本地 Nano/Standard 模型（或 Realtime 模型自身），职责：
   a. 简单问题/闲聊 → 直接回答（低延迟）
   b. 识别到任务意图 → 调用工具/子 Agent（agent.call:<id>），如
      "把这份录音纪要存进知识库""让审查助手过一遍第三章"
   c. 判定为复杂问题（意图分类：需要深度推理/长答案）→ 调用内置工具
      deep.answer(question, context) —— 即"强模型接管"：
        · 立即用 TTS 播一句过场（"这个问题我想一下，稍等"）
        · 后台把问题+必要上下文发给远程强模型（走 ModelRouter 降级链）
        · 强模型流式返回 → 快脑不改写、逐句直送 TTS 播出
  ↓
回复文本 → 口语化后处理（见 18.3）→ TTS → 播放
全程转写双轨落库：messages 表存 (audio_uri, transcript, speaker, latency_ms)
```

要点：**"强模型接管"就是一个工具**（`deep.answer`），不需要新机制——快脑负责实时性与调度，强模型负责质量，二者通过既有 ToolBus 协作。同理，"语音调用智能体行动"就是 `agent.call:*` 工具在语音链路上的自然使用，Run 轨迹照常记录、照常可回放。

### 18.3 语音态 Prompt 与 TTS 净化

- 语音会话注入附加 system 约束：口语化、单次回答 ≤3 句、不用列表/代码/Markdown、数字与单位读法规范（"三点五公里"而非"3.5km"）。
- TTS 前置净化器：剥离 Markdown 符号、URL 缩读、表情移除、中英混排加空格、长数字分组；代码/表格类内容改播"已在屏幕上展示"并推送到聊天流（语音+屏幕双通道）。
- 引用处理：RAG 引用不口播编号，屏幕字幕保留可点引用。

### 18.4 音频工程细则

- 采集：AudioCapturer，16kHz/单声道/PCM16，`SOURCE_TYPE_VOICE_COMMUNICATION`（启用系统 AEC 回声消除——扬声器外放时防自激打断的关键）。
- 播放：AudioRenderer `STREAM_USAGE_VOICE_COMMUNICATION`；蓝牙耳机/车机路由交给系统 audio framework，监听设备切换事件重建流。
- 后台与熄屏：进入电话模式即申请 `continuousTask(AUDIO_RECORDING)`，注册 AVSession（锁屏/控制中心显示通话卡片，可挂断）；退出即释放。
- 权限：麦克风权限按需申请，通话中状态栏常显录音指示（系统强制，合规即可）。
- 功耗预算：级联全本地链路在手机上目标 <8%/h（Nano 模型 + int8 ASR/TTS 可达）；PC 不设限。

### 18.5 电话模式 UI

全屏通话界面（三端同构）：中央声纹动效 + 实时双向字幕流（可关）、静音/挂断/切文本按钮、"当前大脑"指示器（本地/远程/接管中）；PC 端支持最小化为悬浮胶囊窗继续通话。通话结束自动生成会话摘要（复用 §6.2 在线抽取），整通对话与普通会话同权进入记忆管线。

| R1 | §2 native/doc_parsers | 原文写"pdfium/mupdf 二选一"。**MuPDF 是 AGPL-3.0**（商业双授权），一旦链接进来，整个 App 的开源许可将被传染为 AGPL，且对后续任何闭源分发/企业定制构成法律障碍 | **强制选 PDFium（BSD-3-Clause）**，MuPDF 从候选中删除。CI 加许可证扫描防再引入（见 §22.4） |

## 23. 上下文治理：越界压缩（ContextGovernor，core-agent 新增组件）

### 23.1 问题定义

本地模型 ctx 普遍 8k–32k（且 ctx 越大 KV cache 内存越大——手机上 32k ctx 的 4B 模型 KV 就要吃掉 1–2GB），而一次多步 Agent Run 的原始轨迹（RAG 块+工具输出+多轮对话）轻松超过 100k token。**不能等溢出再截断**（丢任务关键信息=任务失败），必须做成主动、分层、有损可控的压缩体系。

### 23.2 令牌预算模型

每个 Run 持有一份预算账本，ContextGovernor 在**每步组 prompt 前**结算：

```
ctx_total（模型上限，本地模型再按 KV 内存预算下调）
  ├─ reserved_output   预留生成空间（maxTokens + 10% 余量）
  ├─ anchor 区（永不压缩）：system + Agent定义 + L3画像 + 任务原始陈述(用户首条)
  │                      + 当前计划(TaskLedger, §24) + 最新用户消息
  ├─ working 区（分级压缩）：会话历史、工具轨迹、RAG 注入
  └─ 水位线：working 占用 >70% ctx → 触发 Compaction；>90% → 紧急降级
```

### 23.3 分级压缩策略（按信息损失从小到大依次启用）

**L1 工具结果外置（无损，默认常开）**
大体积工具输出（RAG 命中原文、file.read、http.fetch 正文）不直接进上下文：全文落 RDB `run_artifacts` 表，上下文中只注入 `摘要头(≤200 token) + artifact:<id>`；同时给 Agent 一个 `artifact.read(id, offset, len)` 工具按需翻页回读。这是最重要的一层——多数"上下文爆炸"其实是工具原文爆炸，外置后损失为零（信息可随时取回）。

**L2 观察折叠（近无损）**
已被后续步骤消化过的历史工具消息，替换为折叠桩：`[step5 kb.search("…") → 已采纳要点：…；全文 artifact:a17]`。折叠由规则触发（距当前 ≥3 步且非 anchor 引用），不动用 LLM，零成本。

**L3 滚动对话摘要（有损，Nano 模型执行）**
会话历史保留最近 K 轮原文（默认 6 轮），更早的滚动并入一段结构化摘要（含：已达成结论、未决问题、用户明确约束——摘要 prompt 强制这三项，防"摘要把约束摘丢"这一经典事故）。摘要任务走本地 Nano，且被 R3 的优先级机制标记为可抢占。

**L4 阶段汇总（有损，长 Run 专用）**
步数 >15 的 Run，按 TaskLedger 的阶段边界把整段轨迹（若干步的 L2/L3 产物）再汇成阶段报告，轨迹树变成"阶段报告 + 当前阶段明细"的两层结构。层级化是无界时长运行的唯一解。

**紧急降级（>90% 水位）**
按序丢弃：过期 RAG 注入 → 折叠桩 → 最老摘要段；仍不够则中止本步，向用户报告"上下文预算耗尽"并给出已有部分结果——**宁可诚实失败，不做静默掐头截尾**。

### 23.4 与 KV cache 的协同（本地推理专项）

- Compaction 会改写 prompt 前缀 → KV 失效 → 下一步全量 prefill（本地 4B 上 8k prefill 约 3–8s，用户可感）。因此：压缩**攒批执行**（水位触发一次压到 50%，而非每步微调），把 prefill 代价从每步摊到每 N 步一次；压缩发生时 UI 显示"整理思路中…"。
- 布局纪律（R2 修订）保证 anchor 区字节级稳定，llama.cpp 开启 prompt cache 复用稳定前缀。
- 每次 Compaction 前，把 working 区中的候选事实（结论/约束/用户偏好）推送一份到 `memory_inbox`——压缩即记忆写入时机，信息从"上下文"降维到"记忆"而不是消失。

## 24. 长时运行（Long-Running Runs）

### 24.1 两类运行的正式区分

| | Interactive Run | **Long Job（新增）** |
|---|---|---|
| 触发 | 用户消息 | 用户显式"后台执行" / Pipeline / （v2 定时触发） |
| 时长 | 秒–分钟 | 分钟–小时–跨天 |
| 前台依赖 | UI 存活 | **无**：App 被杀可恢复 |
| 预算 | 步数 8 / 2min | 步数 200 / token 与费用上限用户设定 |
| 执行体 | 直接跑 | **分段执行器**（见下） |

### 24.2 TaskLedger：把状态从上下文里搬出来

长任务的根本矛盾：任务状态不能寄生在易失的 ctx 里。为每个 Long Job 维护一份结构化台账（RDB，JSON）：

```jsonc
{
  "goal": "…",                       // 不可变
  "plan": [ {"id":"p1","desc":"…","status":"done|active|todo","result_ref":"artifact:x"} ],
  "facts": ["已确认：…"],             // 跨阶段结论
  "open_questions": ["…"],
  "budget": {"steps_used":37,"tokens":…, "cost_cny":…},
  "cursor": {"phase":"p3","step":4}   // 断点
}
```

每步组 prompt 时，注入的不是全部历史，而是 **goal + plan + facts + 当前阶段明细**（天然配合 §23 的 L4 层级）。Agent 通过内置工具 `ledger.update` 修订计划与事实——计划的每次变更都有 oplog，可回放。这个模式使**上下文变成工作集缓存，台账才是真状态**：任意时刻杀掉进程，凭台账 + artifacts 完整恢复。

### 24.3 分段执行器（适配鸿蒙后台模型）

鸿蒙不给无条件常驻后台，Long Job 必须"跑一段、存档、让出"：

```
段（segment）= 连续执行 ≤N 步（默认 10）或 ≤5 分钟
运行时机：
  前台打开 App        → 直接连续跑段
  用户明确"后台执行"   → 申请 continuousTask（数据处理类）跑当前段，段间检查系统压力
  App 被杀 / 无资源    → workScheduler 注册续跑任务（条件可配：仅充电/仅Wi-Fi）
段边界动作（原子事务）：ledger 落盘 + 未完成 LLM 调用作废（下段重发，调用设计为幂等）
用户面：Long Job 卡片（进度=plan 完成度、已耗预算、暂停/终止按钮）；完成/需人工决策时
发系统通知；需要授权的工具调用在后台段中一律挂起转"待确认"，绝不后台静默越权
护栏：R4 循环/停滞检测按段结算；费用上限触顶硬停；连续 2 段零进度自动暂停并通知
```

### 24.4 对既有章节的接线

- §5.2 循环第 1 步的 ContextBuilder 前移交给 ContextGovernor 统一结算（Interactive 与 Long Job 同一套治理，只是预算档不同）。
- §5.3 Agent-as-Tool 补充：子 Agent 以**独立上下文**运行，只回传结论（≤512 token 规约）——委托即隔离，是控制父上下文规模的第一手段，优先于任何压缩。
- §10 指标补充：ctx 峰值占用、Compaction 次数/耗时、prefill 重放时间、Long Job 段数与恢复成功率。
- §12 排期：ContextGovernor（L1/L2/L3）并入 M3（+1 周）；TaskLedger+分段执行器为 **M8（2 周）**，总排期 21→24 周。L4 阶段汇总随 M8 交付。

# v1.3：PDF 解析路线复审 · 产品经理视角全案复审（去背景化）

## 25. PDF 提取：多模态模型方案复审

**结论先行：可以用、而且应该用——但作为分层管线中的一档，不是替代品；且无论走哪档，PDFium 都不可移除。**

### 25.1 为什么 PDFium 删不掉

VL 模型"看"PDF 的前提是先把页面**光栅化成图片**——干这件事的就是 PDFium（或等价渲染器）。所以多模态方案的真实形态是"PDFium 渲染 + VL 模型理解"，PDFium 从"文本抽取器"变成"渲染器+文本层探针"，但始终在链路里。BSD-3 许可不构成任何负担。

### 25.2 两种抽取方式的真实成本对比

| | 文本层直取（PDFium） | VL 模型解析（渲染成图→模型输出 Markdown） |
|---|---|---|
| 适用 | 数字原生 PDF（有文本层） | 扫描件、图表/公式/复杂表格页、文本层损坏页 |
| 精确度 | 字符级精确、零幻觉 | 高但存在幻觉与漏行风险（对入库语料是隐患） |
| 结构 | 段落/坐标，表格结构弱 | **表格/公式/版面还原为 Markdown，RAG 质量上限更高** |
| 端侧成本 | 毫秒级/页，可忽略 | 2B–7B VL 模型 CPU 推理约 **10–60 秒/页**，百页文档=小时级+显著耗电 |
| 远程成本 | — | 快且质量最高，但按页计费 + 语料出端（受隐私围栏约束） |

数字原生 PDF 占日常语料大头，对它们用 VL 是几个数量级的成本浪费；扫描件和复杂版面页不用 VL 又根本读不出来。**所以答案不是二选一，是逐页路由。**

### 25.3 修订后的文档解析管线（替换 §4.2 第 [2] 步的 PDF 分支）

```
PDF → PDFium 逐页探针：文本层字符密度 + 图像占比 + 表格线特征
  ├─ 文本页        → PDFium 直取（默认，零成本）
  ├─ 简单扫描页    → 轻量 OCR 档：PaddleOCR（ONNX 模型经 onnxruntime 端侧运行，
  │                  Apache-2.0，中文识别强，~1–3 秒/页）——纯文字扫描件的性价比档
  ├─ 复杂页（表格/公式/图文混排）→ VL 档：
  │     本地：Qwen-VL 2B/7B GGUF（llama.cpp mtmd，慢，适合零星页）
  │     远程：OpenAI 兼容 VL API（快，整本扫描件的现实选择；受 local_only 围栏硬拦截）
  └─ 产出统一为带结构 Markdown + 页码锚点 → 进入切片流水线
逐页路由由用户可见可改（导入预览界面标注每页将走哪档，可整体强制某档）；
VL/OCR 档产物标记 provenance=ocr|vlm，检索引用时 UI 提示"由 AI 识别，可能有误"并可跳原图核对。
```

许可影响：新增 PaddleOCR（Apache-2.0）、Qwen-VL 权重（Apache-2.0，运行时下载），审计表全绿，无新增风险。排期影响:文本页直取保持 M2；OCR/VL 档整体归入 V2 期（见 §26.4 重切后的路线图）。

- **同质化**："大模型聊天客户端"红海，本产品的反同质化资产是记忆整理引擎+电话模式+三端一多，聊天本身不构成壁垒。

| **V1.5 亮点版** | +6 周 | Must：级联语音链路+电话模式+两级大脑；Should：远程 Realtime 档 | 传播素材（"鸿蒙上给本地模型打电话"）|
| **V2 深水区** | 数据驱动 | 多智能体 Pipeline/Router、Long Job（TaskLedger+分段执行）、PDF OCR/VL 档、视觉输入、端云同步、HMAF 接入 | 由 V1.x 留存用户的行为数据决定优先级 |

裁剪理由：多智能体编排与 Long Job 是重度用户功能，在没有留存数据前属于过度建设（工程设计保留在 §5/§24，实施后置）；语音是最强获客亮点但工程最重，放 V1.5 让前两版先验证根本假设——"有人要在鸿蒙上用自己的模型"。

### 26.5 北极星指标与关键漏斗

- **北极星**：周活跃深度用户（当周完成 ≥3 次多轮会话的用户数）。
- **激活定义**：安装后 24h 内完成"任一模型可用 + 首次成功问答"。**最大流失点预判是本地模型下载**（GB 级），对策：首启提供"极小模型秒级体验档（<500MB）+ 远程 API 三分钟接入向导"双通道，大模型后台续传。
- **待验证的留存假设**：建立过知识库或开启记忆的用户，30 日留存显著高于纯聊天用户（决定 V1.0 资源是否加码记忆/知识库）。
- 反指标：Token 消耗、会话时长不设为目标（与用户价值不对齐）。

### 26.6 分发与开源策略

应用市场（纯血鸿蒙原生应用稀缺期=编辑推荐与活动位的现实机会，主动申报）+ 开源即 GTM：GitHub/Gitee 双仓，开源带来隐私主张的可信度背书、企业用户自编译分发通道、以及贡献者生态（Provider/解析器/工具的插件化架构天然适合接 PR）。DCO+良好 first-issue 治理从 V0.9 起步。

### 26.7 产品风险（区别于 §13 工程风险）

| 风险 | 对策 |
|---|---|
| 端侧模型体验低于用户被云端大模型养高的预期 | 预期管理产品化：模型档位跑分页、"本机能跑什么"直白呈现；默认路由把重活给远程 |
| 窗口期内竞品移植进入 | V0.9 八周上架是硬约束；差异化押在记忆/语音/主权而非聊天 |
| **生成式 AI 合规备案**：中国《生成式人工智能服务管理暂行办法》对"向公众提供服务"的适用边界 | 本产品形态（纯端侧推理+用户自配 API，开发者不运营模型服务）的备案义务边界**必须在 V0.9 上架前取得法务结论**——这是发布前置项，不是可延后项 |
| 开源后被套壳抢发 | Apache-2.0 下无法禁止；靠商标（名称/Logo 注册）+更新节奏+社区正统性防御 |

# v1.4：长程 Agent 上下文压缩专项深化（§27，深化并部分修订 §23/§24）

§23 给了压缩的分层机器，§24 给了状态外置。但**长程**（数十到数百步、跨段、跨天）场景下，压缩本身会成为新的错误源——本节针对"压缩在长程下如何不把任务压坏"补齐机制。

## 27.1 长程压缩的失效模式（设计必须逐一有解）

| 失效模式 | 表现 | 后果 |
|---|---|---|
| F1 约束丢失 | 用户约束（"预算一万以内""不要动A方案"）在两三轮摘要后消失 | Agent 后期违约，任务成果作废 |
| F2 实体腐蚀 | 数字、编号、路径、ID 在摘要改写中被近似化或写错 | 静默错误，最难发现 |
| F3 摘要漂移 | 摘要的摘要（L3→L4）逐层失真，任务后期依据的是"三手转述" | 方向性跑偏 |
| F4 目标漂移 | 上百步后计划演化偏离原始目标而无人察觉 | 做完了别的事 |
| F5 污染固化 | 某步幻觉产物被摘要收编、再被 TaskLedger 收编，从"一次错误"升级为"持久错误状态" | 错误反复复发且自我强化 |
| F6 中部迷失 | 长上下文中间信息注意力衰减，虽在 ctx 内但等效丢失 | 压缩没超限也失效 |

## 27.2 六项对应机制

**M1 可逆压缩：压缩只作用于工作集，永不作用于事实源（对 F2/F3）**
`run_steps`/`run_artifacts` 永远保留全量原始轨迹（RDB 已有，零增量成本）。每个折叠桩/摘要段强制携带溯源指针 `(step_range, artifact_ids)`，并给 Agent 增设 `trace.read(range)` 工具——模型自己发现"摘要信息不够用"时可以**回读原文**。这把压缩从"不可逆信息销毁"变成"可逆的缓存淘汰"（MemGPT 式分页思想）：L1 的 artifact.read 管工具原文，trace.read 管自身历史，两者合起来保证"上下文里没有 ≠ 系统里没有"。

**M2 约束寄存器 + 压缩保真门（对 F1/F2）**
- Run 启动时用 Nano 模型从任务陈述中抽取**约束寄存器**：硬约束/禁止项/验收标准，结构化存 TaskLedger，钉死在 anchor 区（§23.2），任何压缩不得触碰；后续对话新增的约束由 `ledger.update` 增量入册。
- 每次 L3/L4 摘要产出后过**保真门**：a) 实体核对——正则抽取压缩前文本中的数字/编号/路径/URL 集合，要求在摘要或其溯源桩中可寻回，丢失即把该实体强制附录到摘要尾部（机械保真，不靠模型自觉）；b) 约束核对——摘要连同约束寄存器给 Nano 模型做一次"是否存在冲突/遗漏"判定。门不过 → 换更保守档重压（多留原文）。

**M3 目标复诵（recitation，对 F4/F6）**
每步 prompt 的**尾部**（最靠近生成位置、注意力最强处）固定重写一小节：`原始目标（原文引用，不允许改写）+ 当前阶段 + 本步要做什么 + 剩余预算`。这是对抗中部迷失和目标漂移的最廉价手段——不是新增信息，是把关键信息搬到注意力黄金位。长程 Run 每 10 步追加一次显式自检步："当前计划是否仍服务于原始目标？偏离则说明理由并修订计划（进 oplog）"。

**M4 事实需带证据，台账防污染（对 F5）**
TaskLedger 的 `facts[]` 升级为 `{text, evidence: artifact/step 指针, verified: bool}`。规则：**无证据指针的陈述不得入册为事实**，只能入 `open_questions`；整理/汇总环节遇到与证据冲突的"事实"以证据为准。等价于把 §6 记忆系统的溯源纪律（source_refs）复制到任务态——幻觉可以出现在某一步，但很难在没有证据的情况下固化为持久状态。

**M5 压缩与分段执行对齐（对成本，修订 §23.4/§24.3 的衔接）**
长程 Run 每个段边界本来就要全量重预填（进程可能已死，KV 不可延续）——所以**把攒批 Compaction 统一安排在段边界执行**：压缩的 prefill 代价被段重启天然吸收，段内则只做零成本的 L1/L2。规则改写为：段内水位 >85% 才允许段中紧急压缩，否则一律推迟到段界。附带收益：段界压缩后的"阶段报告"恰好就是恢复现场所需的最小上下文，一份产物两用。

**M6 子代理隔离合同（对总量，强化 §5.3/§24.4）**
长程任务中高吞吐、探索性的工序（读长文档、批量检索、穷举比对）**强制**下放子 Agent，且以显式合同约束边界：入参 brief ≤512 token（目标+约束子集+期望产出格式），回传报告 ≤512 token + artifact 指针；子 Agent 上下文用完即弃，父上下文只接收蒸馏物。委托优先于压缩：能不进父上下文的信息，好过进来再压。

## 27.3 压缩质量的回归测试（进 §10 测试策略）

构造"长程记忆保真"金标：脚本化生成 50+ 步合成 Run，在早期步骤植入若干关键事实/约束（含数字与编号类），强制触发多轮 L3/L4 压缩与 ≥2 次段重启后，向 Agent 提问植入项。验收：约束召回 100%（F1 零容忍，靠 M2 机械保障）、实体召回 ≥95%、经 trace.read 补救后 100%。该套件纳入 CI，压缩策略每次改动跑回归——压缩逻辑是最容易"改一处坏三处"的组件，没有回归套件不许合并。

## 27.4 参数默认值（可配，给出出厂值）

```
anchor 预算占比 ≤25% ctx（超出说明 Agent 定义过肥，编辑器告警）
近窗原文轮数 K：Interactive=6，Long Job=3（长程更依赖台账，原文窗口反而收窄）
L3 触发水位 70% / 段中紧急线 85% / 目标压缩后占用 ≤50%
复诵节 ≤200 token；子代理合同 512/512；trace.read 单次回读 ≤2k token
保真门实体核对：\d{2,} 、大写字母数字混合串、路径/URL 三类正则起步
```

一句话总结本节：**长程场景下,压缩的正确性比压缩率重要一个数量级——所有有损操作必须可逆（M1）、可验（M2/27.3）、有锚（M3）、有据（M4），成本则靠与分段执行对齐（M5）和委托隔离（M6）消化。**
