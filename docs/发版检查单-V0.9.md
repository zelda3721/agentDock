## 4b. 摄取链路对抗式审查（T0.9-26，2026-07-15）

对本轮 embed 链路的 4 处改动跑了多镜头对抗式代码审查（native 内存 / 初始化并发 /
正确性语义，逐条 verdict 复核）。结论与处置：

| 发现 | verdict | 处置 |
|---|---|---|
| 清扫 UPDATE 与 `queue.start()` 续跑抢同一行——把正在续跑的文档翻成「失败·点重试」，用户重试触发 `reparse` 的 `processDocument` 与续跑任务并发 `commitChunks`，交错写坏 chunk/vec_ref | CONFIRMED（中） | ✅ 已修：① 清扫先于就绪门闩解锁 + EntryAbility 在 `queue.start()` 前 `await whenReady()`（清扫严格先于续跑）；② `processDocument` 加进程内在飞闸（同 docId 同时只跑一个，双摄取被直接挡下） |
| `whenReady()` 在 fire-and-forget 清扫前解析——门闩解开瞬间 UI 可读到上次会话 stale 'embedding' | PARTIAL（低，自愈 ≤2s） | ✅ 同上修复顺带消除（就绪门闩改为清扫后解锁） |
| 换 embedding 模型的 DIM_MISMATCH 恢复分支不可达（`ensureOpen` 在 try 外，异常越过恢复冒到上层） | CONFIRMED（低，V0.9 仅一款向量模型故不可触发） | ✅ 已修：`openForDim`/`resetForDimChange` 让开库阶段的 DIM_MISMATCH 也触发重置 |
| Float32Array sub-view 时元素数换算过大 | REFUTED（当前所有调用点均整buffer整视图，无触发路径） | 保留现状 + 注释声明约束 |
| 部分 embed 失败 → 文档仍 'ready' 但永久 FTS-only（尤以「先导入后下模型」触发） | CONFIRMED（本机 10k 压测库即此态） | 见下「已知挂账」；诚实降级（FTS 可检索是真话），自动重嵌随 T1.0 |

## 5. 已知挂账（发版说明如实披露）

- PDF 导入不支持（T0.9-12 挂账：PDFium OHOS 交叉编译；picker 不放行 + 诚实报错）
- xlsx/pptx/html 不支持（V1.0 Could）
- **先于本地 embedding 模型导入的文档永久 FTS-only**（`vec_ref` 留 NULL；诚实降级——
  FTS 可检索，但无向量路增强）。恢复途径：手动「重新解析」该文档即重嵌；模型就绪时
  自动重嵌全库 FTS-only 文档 = KB_REINDEX_LIB（T1.0）。审查确认，如实披露。
- 换 embedding 模型 = 库向量重置，老文档退回 FTS 单路（整库重 embed 任务随 T1.0）
- 删除文档留孤儿向量（检索侧如实丢弃；压实随 T1.0 重建任务）

  留待 T1.0-06/07/08；③ 记忆条目默认 local_only（远程 Provider 下召回为空，红线 17）——
